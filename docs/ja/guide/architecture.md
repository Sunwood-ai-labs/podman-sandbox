# アーキテクチャ比較

PodmanとDockerの基本的なアーキテクチャの違いを理解します。

## 概要

| 項目 | Docker | Podman |
|------|--------|--------|
| アーキテクチャ | クライアント・サーバー（デーモン） | デーモンレス（fork-exec） |
| rootアクセス | デフォルトで必要 | デフォルトで不要 |
| セキュリティモデル | 中央デーモン | 直接プロセス実行 |

## デーモンモデル

### Docker

Dockerはクライアント・サーバーアーキテクチャを使用し、中央デーモン（`dockerd`）が：
- すべてのコンテナ操作がデーモンを経由する
- デフォルトでrootとして実行される
- 単一障害点となる

```
┌─────────┐     ┌─────────┐     ┌──────────┐
│  CLI    │────▶│ Daemon  │────▶│ Container │
└─────────┘     └─────────┘     └──────────┘
```

### Podman

Podmanはデーモンレスアーキテクチャを使用：
- CLIが直接コンテナと対話する
- 中央プロセスが不要
- 各ユーザーが独自のコンテナを持つ

```
┌─────────┐     ┌──────────┐
│  CLI    │────▶│ Container │
└─────────┘     └──────────┘
```

## セキュリティへの影響

### Docker
- デーモンが高い権限で実行される
- デーモン経由の攻撃が可能
- 多くの操作にrootアクセスが必要

### Podman
- 特権デーモンプロセスがない
- 直接実行により攻撃対象が減少
- rootlessコンテナがデフォルト
- ユーザー間の分離が良い

## パフォーマンスの考慮

| シナリオ | Docker | Podman |
|----------|--------|--------|
| アイドル時のメモリ | 高い（デーモン実行中） | 低い（デーモンなし） |
| 起動時間 | やや速い（デーモンがウォーム） | やや遅い（新規起動） |
| マルチテナント | デーモン共有 | プロセス分離 |

## どちらを使うべきか

### Dockerを選ぶ場合
- 大規模なコンテナオーケストレーション
- 既存のDockerエコシステムへの投資
- Docker SwarmやDocker Composeワークフロー

### Podmanを選ぶ場合
- 開発環境
- セキュリティ重視のデプロイメント
- systemd統合が必要
- マルチテナントシステム
- リソース制約のある環境
